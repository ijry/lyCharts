# ECharts Runtime API Compatibility Phase 2 Design

**Date:** 2026-07-18

**Scope:** Extend the existing ECharts-style runtime compatibility layer with three focused API groups: legend selection, non-candlestick `dataZoom` for axis charts, and basic `tooltip.formatter` support. Keep behavior aligned across `src/uni_modules/ly-charts` and `uin-app-x/uni_modules/ly-charts`, and update the sibling documentation repository at `D:\Repos\xyito\open\lyCharts-docs`.

## Goals

- Support `legend.selected` as the initial series visibility state.
- Support `dispatchAction({ type: 'legendToggleSelect' })` to toggle a series by name.
- Support non-candlestick `dataZoom` for line, bar, and scatter charts by clipping rendered data to a visible data window.
- Support basic `tooltip.formatter` string templates.
- Allow JavaScript Vue components to execute function `tooltip.formatter` safely.
- Make uvue components degrade safely when `tooltip.formatter` is a function.
- Keep unsupported ECharts actions safe and non-throwing.

## Non-Goals

- Do not implement a full ECharts legend component model or click-to-toggle legend UI.
- Do not implement `legendSelect`, `legendUnSelect`, `legendAllSelect`, or `legendInverseSelect` in this phase.
- Do not implement slider UI for non-candlestick `dataZoom`.
- Do not implement `dataZoom` filtering modes beyond a practical visible-window clip.
- Do not implement HTML tooltip rendering, rich text fragments, async formatter output, or DOM positioning.
- Do not expand this phase to pie, radar, gauge, or candlestick beyond behavior required to remain compatible with existing runtime APIs.

## Current Context

Phase 1 added runtime helpers and exposed instance APIs across Vue and uvue chart components. Current limitations relevant to this phase:

- `dispatchAction({ type: 'dataZoom' })` only changes candlestick charts.
- `legend.selected` is not applied when rendering axis-chart series.
- `dispatchAction({ type: 'legendToggleSelect' })` is not handled.
- Tooltip text is generated by chart-local default logic and ignores `tooltip.formatter`.
- line, bar, and scatter already store chart geometry in `seriesData`, so data-window clipping can be implemented before drawing without a new render pipeline.

## Proposed API Contract

### Legend Selection

Charts should accept:

```js
{
  legend: {
    selected: {
      Sales: true,
      Profit: false
    }
  },
  series: [
    { name: 'Sales', type: 'line', data: [120, 200] },
    { name: 'Profit', type: 'line', data: [30, 80] }
  ]
}
```

Behavior:

- `legend.selected[name] === false` hides the matching series from drawing, hit testing, tooltip aggregation, and data range calculation.
- Missing selected entries default to visible.
- The legend itself is still drawn using the existing helper. This phase does not add dimmed legend styles.
- Runtime toggles update the current normalized option and redraw the chart.

Supported action:

```js
chart.dispatchAction({ type: 'legendToggleSelect', name: 'Sales' })
```

Return value:

- `true` when a named series is found and toggled.
- `false` when the action is invalid, missing `name`, or the chart has no matching series.

### Axis-Chart `dataZoom`

Line and bar charts should support:

```js
chart.dispatchAction({ type: 'dataZoom', start: 20, end: 80 })
chart.dispatchAction({ type: 'dataZoom', startValue: 3, endValue: 12 })
```

Scatter charts should support the same action by clipping data by point index order, not by numeric x-range. This keeps the behavior predictable with the existing renderer and avoids introducing a new scatter axis scale model.

Option-level initial `dataZoom` should also be applied:

```js
{
  dataZoom: [{ start: 20, end: 80 }],
  xAxis: { data: ['A', 'B', 'C', 'D', 'E'] },
  series: [{ type: 'bar', data: [10, 20, 30, 40, 50] }]
}
```

Behavior:

- The visible window is resolved from `start` / `end` percentages or `startValue` / `endValue` indexes.
- Indexes are clamped to the available data length.
- `end` is inclusive after conversion to an index.
- Line/bar `xAxis.data` and every series data array are sliced to the visible window before range calculation and drawing.
- Scatter series data arrays are sliced to the visible window before range calculation and drawing.
- `getOption()` returns the current option with the latest `dataZoom` values.
- No non-candlestick slider UI is drawn in this phase.

### Tooltip Formatter

Support string templates on Vue and uvue:

```js
tooltip: {
  formatter: '{b}: {c}'
}
```

Supported placeholders:

- `{a}`: series name.
- `{b}`: data name / category label.
- `{c}`: value.
- `{d}`: percent when present in the payload, otherwise an empty string.
- `{seriesName}`, `{name}`, `{value}` as readable aliases.

Axis tooltip payloads with multiple series should apply the template per series and join lines with the existing tooltip box rendering. If the formatter string contains newline characters, split them into multiple tooltip lines.

Vue components should also support function formatters:

```js
tooltip: {
  formatter(params) {
    return `${params.name}: ${params.value}`
  }
}
```

Behavior:

- For single-point tooltip payloads, pass the current pointer payload object.
- For axis tooltip payloads with `seriesValues`, pass an array of ECharts-like parameter objects.
- Convert returned values to strings.
- Catch formatter errors, log a concise warning, and fall back to default tooltip content.
- uvue components cannot reliably execute JavaScript functions in all runtimes, so function formatters are ignored and the default tooltip text is used.

## Architecture

### Shared Runtime Helpers

Extend runtime helpers rather than duplicating behavior in each chart:

- JavaScript helper:
  - Resolve legend selected state.
  - Filter visible series.
  - Resolve dataZoom windows.
  - Clip option data by visible window.
  - Format tooltip text from string or function formatter.
- UTS helper:
  - Mirror legend filtering and dataZoom clipping with UTS-compatible syntax.
  - Support string tooltip templates.
  - Treat function formatters as unsupported and return default tooltip lines.

Chart components should call helper functions before calculating data ranges and geometry. The original normalized option should remain the source of truth, while the clipped render option is used for drawing.

### Component Integration

For line, bar, and scatter in both Vue and uvue:

1. Normalize the incoming option as phase 1 already does.
2. Apply `legend.selected` to determine visible series.
3. Resolve the active `dataZoom` window from the option or runtime action state.
4. Build a clipped render option for drawing.
5. Use the render option for x-axis processing, data range calculation, drawing, hit testing, and tooltip aggregation.
6. Keep `currentOption` updated with runtime mutations so `getOption()` reflects legend and dataZoom state.
7. Add `legendToggleSelect` and non-candlestick `dataZoom` branches to `dispatchAction()`.
8. Route tooltip line generation through shared formatter helpers.

Other chart components should keep returning `false` for non-supported `dataZoom` and `legendToggleSelect` unless a chart-local implementation already exists. Existing candlestick `dataZoom` behavior remains unchanged.

## Error Handling

- Invalid `legendToggleSelect` actions return `false`.
- Invalid `dataZoom` payloads return `false`.
- Empty data after clipping should render safely as an empty chart or the existing empty state.
- Formatter exceptions should not break pointer interaction.
- Unsupported formatter functions in uvue should not throw.
- Unsupported action types continue to return `false`.

## Testing

Manual and static validation should cover:

- `legend.selected` hides line, bar, and scatter series on first render.
- `legendToggleSelect` toggles a visible series off and back on.
- Hidden series are excluded from tooltip aggregation and axis range calculation.
- `dataZoom` option clips initial line/bar x-axis data and series data.
- `dispatchAction({ type: 'dataZoom' })` updates visible line/bar/scatter data without throwing.
- Invalid `dataZoom` windows are clamped.
- Tooltip string templates render `{a}`, `{b}`, `{c}`, `{seriesName}`, `{name}`, and `{value}`.
- Vue function formatters receive single-object and array payloads where appropriate.
- uvue function formatters fall back to default tooltip text.
- `git diff --check` passes.
- Existing `pnpm type-check` baseline is recorded if unrelated config/type-definition errors remain.

## Acceptance Criteria

- The three requested API groups are available on line, bar, and scatter where applicable.
- Existing phase 1 runtime APIs continue to work.
- Existing candlestick `dataZoom` behavior is not regressed.
- Vue and uvue behavior stays aligned except for the documented uvue formatter-function limitation.
- Sibling docs accurately describe the new support boundaries.
